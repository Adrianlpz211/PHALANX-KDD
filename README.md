## The problem

Every AI coding tool today has the same fundamental flaw: **the agent forgets everything when you close the chat.**

Session 1: the agent explores your codebase, learns that touching `auth.ts` breaks the session manager, figures out your conventions, understands your architecture.  
Session 2: it knows nothing. It explores again. It makes the same mistake.

This isn't a model problem. It's an architecture problem. No amount of context window size fixes a system that starts from zero every time.

**Agentic KDD solves this at the infrastructure level** ‚ÄĒ with persistent memory that lives inside your project, learns from every cycle, and gets smarter over time.

---

## What is KDD?

**Knowledge-Driven Development** is the methodology at the core of Agentic KDD.

Traditional AI development treats each session as independent. KDD treats knowledge as a first-class artifact: every pattern discovered, every error resolved, every architectural decision made is stored in a structured knowledge graph and fed back into every future cycle.

The agent doesn't re-learn your project. It already knows it.

```
Without KDD:
  Session 1 ‚Üí agent learns ‚Üí session ends ‚Üí forgotten
  Session 2 ‚Üí start over ‚Üí same mistakes ‚Üí same time lost

With KDD:
  Session 1 ‚Üí agent learns ‚Üí stored in .agentic/memoria.db
  Session 2 ‚Üí agent already knows ‚Üí builds on top of everything
  Session N ‚Üí the system is smarter than it was on day one
```

---

## How it works

Agentic KDD lives inside your project as a `.agentic/` directory ‚ÄĒ a local SQLite database, 25 specialized modules, and a pipeline that runs automatically when you type `aa:`.

```bash
aa: implement JWT authentication with refresh token rotation
```

That single command triggers a complete pipeline without interrupting you:

```
‚Ď† Analyst           reads memory, AST graph, ADRs, specs ‚ÄĒ plans all phases
‚Ď° Build             frontend / backend agents implement within the plan
‚ĎĘ TDD Gate          runs tests, self-heals up to 3√ó automatically
‚Ď£ QA                verifies acceptance criteria
‚Ď§ Preservation Gate checks that verified contracts still pass (no collateral damage)
‚Ď• ag:review         automatic code review against KDD memory
‚Ď¶ Memory            syncs graph, causal edges, specs, observability
‚Ďß Creative Engine   detects improvement opportunities for the next cycle
```

**The developer never types `ag: test`, `ag: review`, or `akdd contracts gate`. They run automatically.**

The critical difference from every other agent framework is step ‚ĎĘ. The TDD gate is a deterministic check in Node.js code ‚ÄĒ not a prompt suggestion. The agent cannot declare tests passing without actually running them.

```
Cursor rules:   "prefer running tests before delivering"
                ‚Üí the agent follows this when it feels like it

Agentic KDD:    if (tests_passing === false) { STOP("gate rejected") }
                ‚Üí the code rejects progress without proof of compliance
```

## Memory architecture

Four persistent memory layers, fully offline, in SQLite, inside your project.

```
.agentic/memoria.db
‚Ēā
‚Ēú‚ĒÄ‚ĒÄ WORKING MEMORY      Active session buffer
‚Ēú‚ĒÄ‚ĒÄ EPISODIC MEMORY     Raw history of every cycle
‚Ēú‚ĒÄ‚ĒÄ SEMANTIC MEMORY     Project entity graph, AST, module relationships
‚ĒĒ‚ĒÄ‚ĒÄ PROCEDURAL MEMORY   Patterns, errors, decisions ‚ÄĒ rules applied every cycle
```

Every node carries a confidence signal updated automatically:

```
LOW    ‚Üí suggestion, not enforced
MEDIUM ‚Üí applied and mentioned in every plan
HIGH   ‚Üí fixed rule, applied on every cycle without exception

Applied ‚Č• 3√ó  +  utility ‚Č• 70%  ‚Üí  auto-promoted to MEDIUM
Applied ‚Č• 7√ó  +  utility ‚Č• 80%  ‚Üí  auto-promoted to HIGH
Unused for 60 cycles             ‚Üí  auto-degraded (temporal decay)
```

Three additional knowledge structures specific to software development:

- **AST Graph** ‚ÄĒ symbols, imports, call graph, PageRank across 12 languages
- **Causal Edges** ‚ÄĒ `caused_failure`, `was_fixed_by`, `tested_by`, `regressed_by`, `verifies`, `protects` ‚ÄĒ bi-temporal, never deleted
- **Knowledge Docs** ‚ÄĒ ADRs, gotchas, conventions from `docs/adr/`

## The five phases

### Phase 0 ‚ÄĒ Harness

Deterministic enforcement gates. `harness.cjs` runs PRE/EXEC/POST checks on every pipeline step. `tdd-gate.cjs` is a Node.js loop that detects your test command, runs it, parses output (Jest, Vitest, Mocha, pytest), and retries up to three times automatically.

### Phase 1 ‚ÄĒ Discernment

`ast-indexer.cjs` builds an AST graph across 12 languages with PageRank scoring. `causal-edges.cjs` tracks what broke what, what fixed what ‚ÄĒ bi-temporally, with full history preserved. Includes semantic pruning to prevent context overflow.

### Phase 2 ‚ÄĒ Knowledge Base

`adr-ingestor.cjs` parses Architecture Decision Records in MADR format into typed graph edges ‚ÄĒ no LLM required. `knowledge-ingestor.cjs` ingests gotchas, conventions, and runbooks.

### Phase 3 ‚ÄĒ Autonomy

`spec-manager.cjs` implements Kiro-style specs with wave execution. `impact-analyzer.cjs` runs pre-change impact analysis ‚ÄĒ CRITICAL, MEDIUM, or LOW severity before touching anything.

### Phase 4 ‚ÄĒ Collaborative Mode

`collab-manager.cjs` syncs the local SQLite memory to a shared Turso database. Multiple developers share one knowledge base.

```
Dev A discovers that touching auth.ts breaks session.ts
                        ‚Üď auto-sync to Turso
Dev B already knows it ‚ÄĒ before touching auth.ts
Dev C joins the team   ‚ÄĒ gets 6 months of project knowledge from day one
```

Team members join with a single invite code ‚ÄĒ no Turso account required.

## Preservation Intelligence Layer

The most critical gap in AI-assisted development: **the agent learns what not to do, but doesn't protect what already works.**

```
Bug A ‚Üí Fix A ‚Üí ‚úÖ OK
Bug B ‚Üí Fix B ‚Üí ‚úÖ OK ‚ÄĒ but login is now broken (collateral damage)
Agent: "I didn't repeat Bug A" ‚ÄĒ true, but irrelevant
```

### Contract Guard

`contract-guard.cjs` maintains a live list of verified behavioral contracts ‚ÄĒ things that must keep working regardless of what changes.

Contracts are auto-generated from passing tests. No developer input required.

```
Promotion path:
  candidate  (< 3 consecutive passes)
  verified   (‚Č• 3 passes, failure rate ‚Č§ 5%)
  protected  (‚Č• 7 passes, failure rate ‚Č§ 2%) ‚ÄĒ never allowed to break
```

**Preservation Gate** runs as step ‚Ď§ of every pipeline. It only checks contracts in the blast radius of the modified files ‚ÄĒ not the full test suite.

```
aa: fix session expiration

Impact Analyzer: auth.ts modified
AST query: which contracts protect auth.ts?
‚Üí AUTH-001, AUTH-004, AUTH-006

TDD Gate: scope tests ‚Üí PASS
Preservation Gate: AUTH-001 ‚Üí FAIL
‚Üí STOP: "protected contract broken ‚ÄĒ AUTH-001: valid login generates session"
```

**Blast radius** ‚ÄĒ before touching any file:

```bash
akdd contracts blast src/auth.ts
‚Üí Contracts at risk: 14
‚Üí Severity: HIGH
‚Üí Protected: 8 | Verified: 6
‚Üí Manual review required before proceeding
```

### Creative Engine

`creative-engine.cjs` detects improvement opportunities during each cycle and proposes them ‚ÄĒ without ever applying them unsolicited at Level 1.

```
Level 0 ‚ÄĒ STRICT          execute exactly what was asked, no suggestions
Level 1 ‚ÄĒ ASSISTED        execute + suggest improvements (default from day one)
Level 2 ‚ÄĒ CREATIVE_CONTROLLED   auto-applies low-risk improvements (auto-elevates at 10+ contracts)
Level 3 ‚ÄĒ EXPLORATORY     maximum inference for undocumented projects (manual only)
```

Guardrails always active regardless of level:
- Never touches PROTECTED contracts
- CRITICAL blast radius ‚Üí blocks any creative action
- Every creative action is audited and reversible
- Successful improvements stored as `creative_wins` in memory

## Memory governance ‚ÄĒ MemCurator

`mem-curator.cjs` runs autonomously every 10 cycles:

- **TTL enforcement** ‚ÄĒ episodic memory > 30 days compressed, > 90 days archived
- **Deduplication** ‚ÄĒ nodes with >92% similarity merged, preserving higher confidence
- **Conflict resolution** ‚ÄĒ superseded rules marked `HISTORICO` with explicit link to replacement
- **Relevance scoring** ‚ÄĒ `S(k) = similarity √ó exp(-őĽ √ó őĒt) √ó log(1 + usage_count)`
- **Node limit** ‚ÄĒ active procedural memory capped at 1,000 nodes

```powershell
irm https://raw.githubusercontent.com/Adrianlpz211/Agentic-KDD/main/install.ps1 | iex
```

```bash
cd my-project
akdd init
```

```
aa: configure
```

**Requirements:** Node.js 18+, Cursor or Claude Code (or any MCP client).

## Updating an existing project

```bash
akdd update
```

Downloads latest modules from GitHub. Schema migrations run automatically. Memory stays intact.

---

## What Agentic KDD is not

It is not a chat interface, an IDE, or a replacement for Git. It does not require internet after setup. It does not send your code anywhere. It does not depend on a SaaS backend.

It is infrastructure. It runs inside your project, travels with your repository, and compounds over time.
